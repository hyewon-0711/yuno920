# 기능 정의서: Play (놀이)

> 대상 사용자: 아이 (6~10세) | 관련 탭: Play | 테마 컬러: Purple `#8B5CF6`

---

## 개요

Play 탭은 아이가 직접 참여하는 학습형 놀이 공간으로, 게임을 통해 DAU를 높이고 게임 결과를 성장 분석(hexagon_scores)에 반영하는 것이 핵심 목적이다.

---

## PLAY-01: 계산 게임

### 기본 정보

| 항목 | 내용 |
|------|------|
| 기능 ID | PLAY-01 |
| 기능명 | 계산 게임 |
| 목적 | 사칙연산 훈련을 게임화하여 수학적 사고력 향상 및 매일 접속 동기 부여 |
| 우선순위 | P1 (Sprint 3) |

### 사용자 스토리

- 아이로서, 나는 내 수준에 맞는 계산 문제를 풀고 싶다 → 너무 쉽거나 어렵지 않게
- 아이로서, 나는 문제를 맞히면 점수가 올라가고 레벨이 오르는 것을 보고 싶다
- 부모로서, 나는 아이의 계산 게임 결과가 성장 분석에 반영되는 것을 원한다

### 입력 / 출력

| 구분 | 내용 |
|------|------|
| 입력 | 게임 시작 요청, 문제별 정답 입력 (숫자 키패드) |
| 출력 | 문제 10개, 정답/오답 피드백, 최종 점수, 레벨 변동, 소요 시간 |

### UI 요소

| 요소 | 설명 |
|------|------|
| 게임 선택 카드 | Play 탭 메인에서 🔢 아이콘, 현재 레벨 표시 |
| 게임 화면 | 문제 표시 영역, 숫자 키패드, 진행률 (1/10), 타이머, 점수 |
| 결과 화면 | 총점, 정답/오답 개수, 레벨 변동 애니메이션, 다시하기 버튼 |
| 정답 피드백 | ✅ 초록 애니메이션 + 점수 증가 모션 |
| 오답 피드백 | ❌ 빨간 애니메이션 + 정답 표시 (1.5초) |

### 비즈니스 로직 — 게임 규칙

```
1. 게임 시작 시 child의 현재 레벨을 play_logs에서 조회
2. 레벨에 따라 문제 10개를 클라이언트에서 생성
3. 문제당 제한시간: 레벨 1~5 → 없음, 레벨 6~10 → 30초, 레벨 11+ → 20초
4. 10문제 모두 완료 후 결과 화면 표시
5. 중도 이탈 시 풀었던 문제까지만 저장
```

### 난이도 시스템

| 레벨 | 연산 유형 | 숫자 범위 | 예시 |
|------|-----------|-----------|------|
| 1~3 | 덧셈 | 1~20 | 7 + 5 = ? |
| 4~5 | 덧셈 + 뺄셈 | 1~50 | 32 - 18 = ? |
| 6~8 | 덧셈 + 뺄셈 + 곱셈 | 1~50, 구구단 | 7 × 8 = ? |
| 9~11 | 사칙연산 | 1~100 | 84 ÷ 12 = ? |
| 12~15 | 사칙연산 + 혼합 | 1~200 | (15 + 7) × 3 = ? |
| 16~20 | 혼합 연산 | 1~500 | 125 × 4 - 200 = ? |

**난이도 자동 조절 알고리즘:**

```
최근 3회 게임의 정답률을 계산:
- 정답률 ≥ 90% → 레벨 +1 (최대 20)
- 정답률 ≥ 70% → 레벨 유지
- 정답률 < 70% → 레벨 -1 (최소 1)

조회 쿼리:
SELECT score FROM play_logs
WHERE child_id = {child_id}
  AND game_type = 'math'
ORDER BY played_at DESC
LIMIT 3
```

### 점수 계산 로직

```
기본 점수: 정답 1개 = 10점
시간 보너스: 
  - 문제당 5초 이내 정답 → +3점
  - 문제당 10초 이내 정답 → +1점
연속 정답 보너스:
  - 3연속 → +5점
  - 5연속 → +10점
  - 10연속 (퍼펙트) → +30점
최종 점수 = 기본 + 시간 보너스 + 연속 보너스
만점 = 10×10 + 10×3 + 5 + 10 + 30 = 175점
```

### 예외 처리

| 상황 | 처리 |
|------|------|
| 첫 게임 (play_logs 없음) | 레벨 1로 시작 |
| 네트워크 끊김 (게임 중) | 로컬 저장 후 복구 시 동기화 |
| 문제당 시간 초과 | 오답 처리, 정답 표시 후 다음 문제 |
| 게임 중 앱 백그라운드 | 타이머 일시정지, 복귀 시 재개 |
| 레벨 최대(20) 도달 | 레벨 유지, "최고 레벨 달성!" 배지 표시 |

### 관련 테이블

| 테이블 | 용도 | 필드 |
|--------|------|------|
| `play_logs` | 게임 결과 저장 | id, child_id, game_type='math', score, level, detail_json, played_at |
| `hexagon_scores` | 학습 영역 점수 반영 | child_id, category='학습', score, updated_at |

### 관련 API

| Method | Endpoint | 설명 |
|--------|----------|------|
| GET | Supabase Client → `play_logs` | 현재 레벨 및 최근 기록 조회 |
| POST | Supabase Client → `play_logs` | 게임 결과 저장 |
| PATCH | Supabase Client → `hexagon_scores` | 학습 영역 점수 갱신 |

## PLAY-02: 기억력 게임

### 기본 정보

| 항목 | 내용 |
|------|------|
| 기능 ID | PLAY-02 |
| 기능명 | 기억력 게임 |
| 목적 | 카드 매칭 게임을 통한 기억력/집중력 향상 |
| 우선순위 | P1 (Sprint 3) |

### 사용자 스토리

- 아이로서, 나는 같은 그림의 카드 짝을 맞추는 게임을 하고 싶다
- 아이로서, 나는 레벨이 올라갈수록 카드가 많아지는 도전을 즐기고 싶다
- 아이로서, 나는 빠르게 맞출수록 높은 점수를 받고 싶다

### 입력 / 출력

| 구분 | 내용 |
|------|------|
| 입력 | 게임 시작 요청, 카드 터치 (뒤집기) |
| 출력 | 카드 그리드, 매칭 결과 애니메이션, 최종 점수, 레벨 변동 |

### UI 요소

| 요소 | 설명 |
|------|------|
| 게임 선택 카드 | Play 탭 메인에서 🧠 아이콘, 현재 레벨 표시 |
| 카드 그리드 | 레벨에 따른 크기, 뒤집기 애니메이션 (3D flip) |
| 상단 바 | 시간, 시도 횟수, 남은 짝 수 |
| 매칭 성공 | 카드 하이라이트 → 사라짐 (scale-down) |
| 매칭 실패 | 빨간 흔들림 → 다시 뒤집힘 (0.8초 후) |
| 결과 화면 | 완료 시간, 시도 횟수, 점수, 레벨 변동 |

### 비즈니스 로직 — 게임 규칙

```
1. 게임 시작 시 레벨에 맞는 그리드 크기 결정
2. 카드 이미지는 동물/과일/도형 등 아이 친화적 아이콘 사용
3. 카드를 2초간 전체 공개 → 뒤집기 → 게임 시작
4. 한 번에 2장만 뒤집기 가능
5. 2장이 같으면 매칭 성공 (카드 제거)
6. 2장이 다르면 0.8초 후 다시 뒤집힘
7. 모든 짝을 맞추면 게임 완료
```

### 난이도 시스템

| 레벨 | 그리드 | 카드 수 | 짝 수 | 시간 제한 |
|------|--------|---------|-------|-----------|
| 1~3 | 3×4 | 12장 | 6짝 | 120초 |
| 4~6 | 4×4 | 16장 | 8짝 | 120초 |
| 7~9 | 4×5 | 20장 | 10짝 | 150초 |
| 10~12 | 5×6 | 30장 | 15짝 | 180초 |
| 13+ | 5×6 | 30장 | 15짝 | 120초 (단축) |

**난이도 자동 조절 알고리즘:**

```
최근 3회 게임 결과 평가:
- 3회 모두 시간 내 완료 + 평균 시도 횟수 ≤ (짝 수 × 1.5) → 레벨 +1
- 3회 중 2회 이상 시간 내 완료 → 레벨 유지
- 3회 중 2회 이상 시간 초과 → 레벨 -1 (최소 1)
```

### 점수 계산 로직

```
기본 점수: 매칭 1짝 = 10점
효율 보너스: 시도 횟수가 최소(= 짝 수)에 가까울수록 보너스
  - 시도 횟수 = 짝 수 (퍼펙트) → +50점
  - 시도 횟수 ≤ 짝 수 × 1.3 → +30점
  - 시도 횟수 ≤ 짝 수 × 1.5 → +10점
시간 보너스: 제한 시간의 50% 이내 완료 → +20점
연속 매칭: 3연속 즉시 매칭 → +15점
```

### 예외 처리

| 상황 | 처리 |
|------|------|
| 시간 초과 | 게임 종료, 맞춘 짝까지만 점수 계산, "시간 초과!" 표시 |
| 빠른 연타 (3장 이상) | 2장 뒤집힌 상태에서 추가 터치 무시 |
| 첫 게임 | 레벨 1 (3×4 그리드), 간단한 튜토리얼 표시 |
| 앱 백그라운드 | 타이머 일시정지, 카드 상태 보존 |

### 관련 테이블

| 테이블 | 용도 | 필드 |
|--------|------|------|
| `play_logs` | 게임 결과 저장 | id, child_id, game_type='memory', score, level, detail_json, played_at |
| `hexagon_scores` | 학습 영역 점수 반영 | child_id, category='학습', score, updated_at |

### 관련 API

| Method | Endpoint | 설명 |
|--------|----------|------|
| GET | Supabase Client → `play_logs` | 현재 레벨 조회 |
| POST | Supabase Client → `play_logs` | 게임 결과 저장 |
| PATCH | Supabase Client → `hexagon_scores` | 학습 영역 점수 갱신 |

## PLAY-03: 독서 기반 게임

### 기본 정보

| 항목 | 내용 |
|------|------|
| 기능 ID | PLAY-03 |
| 기능명 | 독서 기반 게임 (독서 퀴즈) |
| 목적 | 독서 내용을 게임으로 복습하여 읽기 효과 강화, 독서 동기 부여 |
| 우선순위 | P1 (Sprint 4) |

### 사용자 스토리

- 아이로서, 나는 최근 읽은 책에 대한 퀴즈를 풀어보고 싶다
- 아이로서, 나는 책 내용을 얼마나 잘 기억하는지 확인하고 싶다
- 부모로서, 나는 아이가 독서 후 내용을 이해했는지 확인하고 싶다

### 입력 / 출력

| 구분 | 내용 |
|------|------|
| 입력 | 책 선택 (최근 읽은 책 목록에서), 퀴즈 답안 선택 |
| 출력 | OpenAI가 생성한 객관식 퀴즈 3~5문제, 정답/해설, 최종 점수 |

### UI 요소

| 요소 | 설명 |
|------|------|
| 게임 선택 카드 | Play 탭 메인에서 📖 아이콘, 풀 수 있는 퀴즈 수 표시 |
| 책 선택 화면 | 최근 읽은 책 목록 (reading_logs), 각 책별 "퀴즈 풀기" 버튼 |
| 퀴즈 화면 | 문제 텍스트, 4지선다 객관식 보기, 진행률 (1/5) |
| 정답 피드백 | ✅ 정답 + 간단한 해설 |
| 오답 피드백 | ❌ 정답 표시 + 해설 |
| 결과 화면 | 맞은 개수/전체, 점수, 책 관련 격려 메시지 |

### 비즈니스 로직 — 게임 규칙

```
1. reading_logs에서 최근 7일 이내 읽은 책 목록 조회
2. 아이가 책을 선택하면 FastAPI로 퀴즈 생성 요청
3. FastAPI가 OpenAI API를 호출하여 퀴즈 생성
4. 퀴즈 유형: 단어 의미, 내용 기억, 등장인물, 줄거리 순서
5. 문제 수: 기본 3문제, 레벨 5+ 시 5문제
6. 객관식 4지선다
7. 퀴즈 완료 후 결과 저장
```

**OpenAI 퀴즈 생성 프롬프트 구조:**

```
시스템 프롬프트:
"당신은 초등학생 아이를 위한 독서 퀴즈 출제자입니다.
아이의 나이({age}세)에 맞는 쉬운 말로 문제를 만들어주세요."

사용자 프롬프트:
"다음 책에 대한 객관식 퀴즈를 {count}개 만들어주세요.
책 제목: {book_title}
책 카테고리: {category}
아이 나이: {age}세

퀴즈 유형:
1. 단어 퀴즈 (주요 단어의 의미)
2. 내용 기억 (주요 사건/장면)

각 문제는 다음 JSON 형식으로:
{
  "question": "문제 텍스트",
  "options": ["보기1", "보기2", "보기3", "보기4"],
  "answer": 0,
  "explanation": "간단한 해설",
  "type": "word" | "content"
}"
```

### 난이도 시스템

| 레벨 | 문제 수 | 문제 유형 | 보기 수 |
|------|---------|-----------|---------|
| 1~4 | 3문제 | 단어 퀴즈 위주 | 3개 |
| 5~8 | 4문제 | 단어 + 내용 혼합 | 4개 |
| 9+ | 5문제 | 내용 기억 + 추론 | 4개 |

### 점수 계산 로직

```
기본 점수: 정답 1개 = 20점
전문 정답: 모두 맞추면 +20점 보너스
레벨 반영: 레벨 5+ 문제 정답 시 1.5배 점수
```

### 예외 처리

| 상황 | 처리 |
|------|------|
| 읽은 책이 없음 (reading_logs 비어있음) | "먼저 책을 읽고 기록해보세요!" 안내, 독서 기록 페이지로 이동 유도 |
| OpenAI API 호출 실패 | "퀴즈를 준비하는 중이에요. 잠시 후 다시 시도해주세요." 리트라이 1회 |
| OpenAI 응답 파싱 실패 | 재요청 (최대 2회), 실패 시 기본 퀴즈 템플릿 사용 |
| 퀴즈 생성 시간 초과 (>10초) | 로딩 애니메이션 + "열심히 문제를 만들고 있어요..." 메시지 |
| 같은 책 재도전 | 이전과 다른 퀴즈 생성 (프롬프트에 "이전 문제와 다르게" 추가) |

### 관련 테이블

| 테이블 | 용도 | 필드 |
|--------|------|------|
| `reading_logs` | 최근 읽은 책 조회 | id, child_id, title, category, read_date |
| `play_logs` | 퀴즈 결과 저장 | id, child_id, game_type='reading_quiz', score, level, detail_json, played_at |
| `hexagon_scores` | 학습 영역 점수 반영 | child_id, category='학습', score, updated_at |

### 관련 API

| Method | Endpoint | 설명 |
|--------|----------|------|
| GET | Supabase Client → `reading_logs` | 최근 읽은 책 목록 조회 |
| POST | `/api/play/generate-quiz` | OpenAI로 퀴즈 생성 (FastAPI) |
| POST | Supabase Client → `play_logs` | 퀴즈 결과 저장 |

## PLAY-04: 게임 결과 / 보상

### 기본 정보

| 항목 | 내용 |
|------|------|
| 기능 ID | PLAY-04 |
| 기능명 | 게임 결과 / 보상 시스템 |
| 목적 | 게임 결과를 성장 데이터에 반영하고, 보상 경험으로 지속적 참여 유도 |
| 우선순위 | P1 (Sprint 3) |

### 사용자 스토리

- 아이로서, 나는 게임이 끝난 후 멋진 결과 화면과 보상을 보고 싶다
- 아이로서, 나는 레벨이 오르면 축하 애니메이션을 보고 싶다
- 부모로서, 나는 아이의 게임 결과가 종합 성장 분석에 반영되길 원한다

### 입력 / 출력

| 구분 | 내용 |
|------|------|
| 입력 | 게임 완료 데이터 (점수, 레벨, 상세 결과) |
| 출력 | play_logs 저장, hexagon_scores 갱신, 보상 애니메이션 |

### UI 요소

| 요소 | 설명 |
|------|------|
| 결과 카드 | 점수 (카운트 업 애니메이션), 정답률, 소요 시간 |
| 레벨 변동 | 레벨 업: 🎉 컨페티 + "레벨 업!" 텍스트 / 레벨 다운: 격려 메시지 |
| 보상 배지 | 특정 조건 달성 시 배지 표시 (첫 게임, 연속 3일, 퍼펙트 등) |
| 액션 버튼 | "다시하기", "다른 게임", "놀이터로" |
| 최근 기록 | Play 탭 하단에 최근 3개 게임 결과 리스트 |

### 비즈니스 로직

**결과 저장 프로세스:**

```
1. 게임 완료 시 play_logs에 결과 INSERT
2. 레벨 자동 조절 알고리즘 실행 (최근 3회 기준)
3. hexagon_scores의 "학습" 카테고리 점수 갱신
4. 보상 조건 체크 → 해당 시 보상 애니메이션 트리거
```

**hexagon_scores 반영 로직:**

```
학습 영역 점수 계산:
  - 최근 7일간 play_logs의 평균 정답률
  - 계산 게임 가중치: 40%
  - 기억력 게임 가중치: 30%
  - 독서 퀴즈 가중치: 30%
  
score = (math_avg × 0.4) + (memory_avg × 0.3) + (quiz_avg × 0.3)
→ 0~100 사이 정규화하여 hexagon_scores에 UPSERT
```

**보상 조건:**

| 배지 | 조건 | 아이콘 |
|------|------|--------|
| 첫 발걸음 | 첫 게임 완료 | 🏅 |
| 꾸준한 도전 | 3일 연속 게임 플레이 | 🔥 |
| 한 주의 왕 | 7일 연속 게임 플레이 | 👑 |
| 계산 마스터 | 계산 게임 퍼펙트 달성 | 🧮 |
| 기억력 챔피언 | 기억력 게임 퍼펙트 달성 | 🧠 |
| 독서 박사 | 독서 퀴즈 전문 정답 | 📚 |
| 레벨 10 돌파 | 아무 게임에서 레벨 10 달성 | ⭐ |
| 만점왕 | 아무 게임에서 최고 점수 갱신 | 🏆 |

### 보상 모션 상세

```
레벨 업 애니메이션:
  1. 결과 카드 표시 (fade-in, 300ms)
  2. 점수 카운트 업 (0 → 최종점수, 1초)
  3. 레벨 업 감지 시:
     - 0.5초 대기
     - 화면 중앙에 "Level Up!" 텍스트 (scale 0→1, bounce)
     - 컨페티 파티클 (2초)
     - 새 레벨 숫자 표시
  4. 배지 획득 시:
     - 레벨 업 후 0.3초 대기
     - 배지 아이콘 팝업 (scale + glow 효과)
     - "첫 게임 달성!" 텍스트

레벨 다운:
  1. 결과 카드 표시
  2. 격려 메시지: "괜찮아! 다음엔 더 잘할 수 있어 💪"
  3. 부드러운 레벨 전환 (flash 없이)
```

### 점수 계산 로직

> PLAY-04 자체 점수는 없음. PLAY-01 ~ PLAY-03의 점수를 수집하여 hexagon_scores에 반영하는 역할.

### 예외 처리

| 상황 | 처리 |
|------|------|
| play_logs 저장 실패 | 로컬에 임시 저장, 다음 앱 접속 시 동기화 시도 |
| hexagon_scores 갱신 실패 | 백그라운드 리트라이 (최대 3회), 실패 시 다음 게임 후 재계산 |
| 게임 데이터 부족 (7일 이내 1회 미만) | 기존 hexagon_scores 유지, 변경하지 않음 |
| 보상 중복 획득 시도 | 이미 획득한 배지는 무시, 중복 트리거 방지 |

### 관련 테이블

| 테이블 | 용도 | 필드 |
|--------|------|------|
| `play_logs` | 모든 게임 결과 저장 | id, child_id, game_type, score, level, detail_json, played_at |
| `hexagon_scores` | 6각형 인재 분석 점수 | id, child_id, category, score, updated_at |

### 관련 API

| Method | Endpoint | 설명 |
|--------|----------|------|
| POST | Supabase Client → `play_logs` | 게임 결과 저장 |
| GET | Supabase Client → `play_logs` | 최근 게임 기록 조회 |
| PATCH | Supabase Client → `hexagon_scores` | 학습 영역 점수 UPSERT |
| POST | `/api/ai/hexagon` | 전체 6각형 점수 재계산 (선택적) |
