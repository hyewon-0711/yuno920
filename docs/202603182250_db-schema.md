## 2. 테이블 설계

### 2-1. `users` — 사용자 프로필

Supabase `auth.users`를 확장하는 public 프로필 테이블. `auth.users.id`와 1:1 매핑된다.

```sql
CREATE TABLE public.users (
  id          UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  email       TEXT NOT NULL,
  name        TEXT NOT NULL,
  avatar_url  TEXT,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

COMMENT ON TABLE  public.users IS '사용자 공개 프로필 (auth.users 확장)';
COMMENT ON COLUMN public.users.id IS 'auth.users.id와 동일한 UUID';
COMMENT ON COLUMN public.users.email IS '이메일 주소 (auth.users.email 사본)';
COMMENT ON COLUMN public.users.name IS '표시 이름';
COMMENT ON COLUMN public.users.avatar_url IS '프로필 이미지 URL (Supabase Storage)';
COMMENT ON COLUMN public.users.created_at IS '가입 시각';
```

**트리거**: 회원가입 시 `auth.users` INSERT → `public.users` 자동 생성

```sql
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO public.users (id, email, name)
  VALUES (
    NEW.id,
    NEW.email,
    COALESCE(NEW.raw_user_meta_data->>'name', split_part(NEW.email, '@', 1))
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_auth_user_created
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_user();
```

### 2-3. `family_members` — 가족 연결

한 아이에 대해 여러 가족 구성원이 역할별로 접근할 수 있다.

```sql
CREATE TABLE public.family_members (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES public.users(id) ON DELETE CASCADE,
  child_id    UUID NOT NULL REFERENCES public.children(id) ON DELETE CASCADE,
  role        family_role NOT NULL DEFAULT 'viewer',
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),

  UNIQUE (user_id, child_id)  -- 같은 사용자가 같은 아이에 중복 등록 불가
);

COMMENT ON TABLE  public.family_members IS '가족 구성원 연결 (사용자 ↔ 아이)';
COMMENT ON COLUMN public.family_members.user_id IS '가족 구성원 (users.id)';
COMMENT ON COLUMN public.family_members.child_id IS '연결된 아이 (children.id)';
COMMENT ON COLUMN public.family_members.role IS '역할: admin(관리자), editor(편집자), viewer(뷰어)';
```

```sql
COMMENT ON COLUMN public.records.user_id IS '기록 작성자';
COMMENT ON COLUMN public.records.content IS '기록 내용 (텍스트)';
COMMENT ON COLUMN public.records.mood IS '감정 상태: happy/neutral/sad/sick/tired';
COMMENT ON COLUMN public.records.categories IS '카테고리 배열 (복수 선택 가능)';
COMMENT ON COLUMN public.records.photos IS '사진 URL 배열 (Supabase Storage, 최대 5장)';
COMMENT ON COLUMN public.records.recorded_at IS '기록 날짜 (실제 사건 날짜)';
COMMENT ON COLUMN public.records.updated_at IS '마지막 수정 시각';
```

**자동 updated_at 갱신 트리거**:

```sql
CREATE OR REPLACE FUNCTION public.update_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = now();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER set_records_updated_at
  BEFORE UPDATE ON public.records
  FOR EACH ROW EXECUTE FUNCTION public.update_updated_at();
```

## 4. RLS 정책

모든 테이블에 RLS를 활성화한다. 접근 권한은 `family_members` 테이블의 role을 기준으로 판단한다.

### 4-0. 공통 헬퍼 함수

```sql
-- 현재 사용자가 특정 아이에 대해 접근 권한이 있는지 확인
CREATE OR REPLACE FUNCTION public.has_child_access(p_child_id UUID)
RETURNS BOOLEAN AS $$
BEGIN
  RETURN EXISTS (
    SELECT 1 FROM public.family_members
    WHERE user_id = auth.uid()
      AND child_id = p_child_id
  );
END;
$$ LANGUAGE plpgsql SECURITY DEFINER STABLE;

-- 현재 사용자가 특정 아이에 대해 쓰기 권한(admin 또는 editor)이 있는지 확인
CREATE OR REPLACE FUNCTION public.has_child_write_access(p_child_id UUID)
RETURNS BOOLEAN AS $$
BEGIN
  RETURN EXISTS (
    SELECT 1 FROM public.family_members
    WHERE user_id = auth.uid()
      AND child_id = p_child_id
      AND role IN ('admin', 'editor')
  );
END;
$$ LANGUAGE plpgsql SECURITY DEFINER STABLE;

-- 현재 사용자가 아이를 직접 생성한 소유자인지 확인
CREATE OR REPLACE FUNCTION public.is_child_owner(p_child_id UUID)
RETURNS BOOLEAN AS $$
BEGIN
  RETURN EXISTS (
    SELECT 1 FROM public.children
    WHERE id = p_child_id
      AND user_id = auth.uid()
  );
END;
$$ LANGUAGE plpgsql SECURITY DEFINER STABLE;
```

```sql
  ON public.users FOR SELECT
  USING (
```

```sql
-- admin만 가족 구성원 추가/수정/삭제 가능
CREATE POLICY "family_members_insert"
  ON public.family_members FOR INSERT
  WITH CHECK (
    EXISTS (
      SELECT 1 FROM public.family_members
      WHERE user_id = auth.uid()
        AND child_id = family_members.child_id
        AND role = 'admin'
    )
    OR public.is_child_owner(child_id)
  );

CREATE POLICY "family_members_delete"
  ON public.family_members FOR DELETE
  USING (
    EXISTS (
      SELECT 1 FROM public.family_members fm
      WHERE fm.user_id = auth.uid()
        AND fm.child_id = family_members.child_id
        AND fm.role = 'admin'
    )
  );
```

## 6. 마이그레이션 순서

FK 의존성을 고려한 실행 순서:

```
Step 1: Enum 타입 생성
  → gender_type, family_role, mood_type, record_category,
    repeat_type, reading_category, game_type, report_type,
    milestone_category

Step 2: 기본 테이블 (의존성 없음)
  → users (+ auth trigger)

Step 3: children (users에 의존)
  → children

Step 4: family_members (users + children에 의존)
  → family_members

Step 5: 데이터 테이블 (children에 의존)
  → schedules
  → records (+ updated_at trigger)
  → milestones
  → reading_logs
  → growth_metrics
  → hexagon_scores
  → play_logs
  → ai_reports

Step 6: notifications (users + children에 의존)
  → notifications

Step 7: 인덱스 생성

Step 8: RLS 정책 적용
  → 헬퍼 함수
  → 각 테이블 RLS enable + 정책

Step 9: 아이 프로필 생성 시 family_member 자동 추가 트리거
```

**아이 프로필 생성 시 admin 역할 자동 부여**:

```sql
CREATE OR REPLACE FUNCTION public.handle_new_child()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO public.family_members (user_id, child_id, role)
  VALUES (NEW.user_id, NEW.id, 'admin');
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_child_created
  AFTER INSERT ON public.children
  FOR EACH ROW EXECUTE FUNCTION public.handle_new_child();
```
